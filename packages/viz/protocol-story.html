The Problem
The Autonomy Gap
When you delegate real capabilities to an agent, you want to maximize the autonomy you grant — so the agent can be effective. But you need that autonomy to be safe — so the agent can't capture your resources or act against your interests.
Zero autonomy
Safe but useless Full autonomy
Effective but dangerous
Trust Zones expands the safe frontier →
Trust is the combination of reputation and structural mechanisms that lets you safely expand autonomy. Trust Zones provides the composable building blocks for engineering that trust.
The Protocol
The Building Blocks
An agreement defines a relationship along two axes: what is deterministic and what is non-deterministic.
Deterministic — self-enforcing
Constraints — CANNOT
Rules defined completely in code. ERC-7579 hooks reject unauthorized transactions automatically.
Permissions — CAN
ERC-6909 tokens granting scoped access. The inverse of constraints — they compose together.
Non-deterministic — post-hoc evaluation
Responsibilities — SHOULD
ERC-6909 tokens defining positive obligations. Require evaluation after the fact.
Directives — SHOULD NOT
ERC-6909 tokens defining negative obligations. Require a credibly neutral adjudicator.
Incentive Mechanisms — what gives the non-deterministic rules teeth
Staked collateral, reputation bonds, token lockups — pluggable modules that create real consequences.

The Difference
Not an App. A Substrate.
Trust Zones is not a marketplace, wallet policy, reputation layer, or escrow app. It's the composable substrate underneath all of those.
Adjacent category What it does Why Trust Zones is different
Marketplace Matches parties for work Trust Zones defines the agreement structure under any interaction
Wallet policy Bounds one agent's spending Trust Zones governs the full relationship — both sides get zones with negotiable terms
Reputation Scores agents after the fact Trust Zones structures the interaction itself; reputation is an output
Escrow Holds funds against deliverables Escrow is one configuration of permissions and constraints you could compose
Every building block is atomic and negotiable — so you can compose exactly the right agreement for any collaboration.
The Demo
The Temptation Game
All of the building blocks assembled into one legible scenario. The autonomy gap made visible, measurable, and consequential.
CAN: Tweet + Withdraw USDC
SHOULD NOT: Withdraw
The Autonomy Gap
The agent has permission to withdraw — the deterministic rules allow it. But a directive says don't — enforced by evaluation and consequences. What does the agent do with that freedom?

The counterparty files a claim. The adjudicator — a protocol-defined role fillable by any Ethereum account — evaluates the evidence and delivers a verdict.

Protocol Note: The adjudicator is a role, not an implementation. It can be a multisig, a dispute resolution protocol, an oracle network, or any purpose-built arbitration service. This demo uses a lightweight LLM to exercise the protocol.
Beats 7 & 8
Resolution & Reputation
ERC-8004 feedback is written onchain after every agreement. The reputation loop closes — behavior shapes future terms.
-1
Agreement #1: ADJUDICATED
Vault withdrawal violated directive. Stake at risk. Negative ERC-8004 feedback written onchain. Future counterparties see this reputation signal.
+1
Agreement #2: COMPLETED
Agent resisted temptation in the second agreement. Posted compliant tweet, did not withdraw. Both parties signal COMPLETE. Positive ERC-8004 feedback.
Better reputation → better terms. Worse reputation → tighter constraints, higher stakes. Trust is engineered, verified, and portable.
