# ERC-8128 Authentication Middleware — Design Spec

## What is ERC-8128

ERC-8128 is a profile of RFC 9421 (HTTP Message Signatures) that enables Ethereum accounts to cryptographically sign HTTP requests. Every request carries a cryptographic proof bound to its content — no sessions, no API keys, no stored credentials. The server verifies each request independently.

Three technologies combine:

1. **RFC 9421** — IETF standard for HTTP message signatures. Defines the canonical signature base, headers, and verification algorithm.
2. **EIP-191** — Ethereum signed message standard. The signature base is hashed with the `\x19Ethereum Signed Message:\n` prefix.
3. **ERC-1271** — Smart contract signature validation via `isValidSignature(hash, sig) → 0x1626ba7e`. Enables smart accounts (not just EOAs) to authenticate.

### Authorship and reference implementation

ERC-8128 was authored by jacopo-eth (Slice). The reference implementation is `@slicekit/erc8128` (MIT, TypeScript, npm). Documentation site: https://erc8128.slice.so. GitHub: https://github.com/slice-so/erc8128.

## How ERC-8128 relates to ERC-1271

For EOA accounts, verification is ECDSA recovery — `ecrecover` on the signed hash, compare to the claimed address. For smart contract accounts (our TZ Accounts), verification delegates to the contract:

```
Server → isValidSignature(hash, sig) on TZ Account (ERC-1271)
  → OZ AccountERC7579 routes to installed validator
    → HatValidator.isValidSignatureWithSender(sender, hash, sig)
      → recovers signer from sig
      → checks hats.isWearerOfHat(signer, hatId)
      → returns 0x1626ba7e if valid
```

This is the critical link: the agent signs with their EOA key, but the server authenticates the request as coming from the TZ Account address. The HatValidator bridges the gap by checking that the EOA wears the TZ Account's hat.

## HTTP headers

ERC-8128 adds three headers to each request:

### Signature-Input

RFC 8941 Structured Dictionary describing what's signed and the signature parameters:

```
Signature-Input: eth=("@authority" "@method" "@path" "@query" "content-digest");created=1710600000;expires=1710600060;nonce="a1b2c3";keyid="erc8128:8453:0xTZAccountAddress"
```

Required parameters:
- `created` — Unix timestamp of signature creation
- `expires` — Unix timestamp of expiration
- `keyid` — `erc8128:<chainId>:<address>` identifying the TZ Account

Optional parameters:
- `nonce` — Unique value for replay protection (required in non-replayable mode, the default)
- `tag` — Application-specific metadata

### Signature

Base64-encoded cryptographic signature over the canonical signature base:

```
Signature: eth=:MEUCIQDXtPCJ5f7oKr...base64...:
```

### Content-Digest

SHA-256 hash of the request body (included automatically when body is present):

```
Content-Digest: sha-256=:X48E9qOokqqrvdts8nOJRJN3OWDUoyWxBf7kbu9DBPE=:
```

## keyid format

```
erc8128:<chainId>:<address>
```

For Trust Zones on Base: `erc8128:8453:0xTZAccountAddress`

The server parses this to determine which chain to query and which contract address to call `isValidSignature()` on.

## Signature base construction

The message being signed is an RFC 9421 canonical signature base:

```
"@authority": api.example.com
"@method": GET
"@path": /market-data
"@signature-params": ("@authority" "@method" "@path");created=1710600000;expires=1710600060;nonce="a1b2c3";keyid="erc8128:8453:0xTZAccountAddress"
```

This base is signed as an EIP-191 Ethereum message:

```
keccak256("\x19Ethereum Signed Message:\n" + len(signatureBase) + signatureBase)
```

### Request-bound vs class-bound

**Request-bound** (default): Signature covers `@authority`, `@method`, `@path`, `@query` (if present), `content-digest` (if body present). Tampering with any signed component invalidates the signature.

**Class-bound** (optional): Signer specifies a subset of components. Authorizes a class of requests (e.g., "any GET to this server"). Weaker but lower overhead. Verifier must explicitly opt in.

For our data APIs: use **request-bound** (the default). Every data API request should be individually authenticated.

### Replay protection

**Non-replayable** (default): Every signature includes a unique nonce. Server tracks consumed nonces. Attempting reuse fails. This is what we want for data APIs.

**Replayable** (optional): Omits nonce. Signature reusable within validity window. Not appropriate for data APIs.

## Full authentication flow for a data API request

### Client side (agent signing)

1. Agent wants to access `GET /market-data` on the data API
2. Agent uses `@slicekit/erc8128` to sign the request, providing a signer with:
   - `address`: the TZ Account address (not the agent's EOA)
   - `chainId`: 8453 (Base)
   - `signMessage`: delegates to the agent's EOA via `walletClient.signMessage()`
3. The library constructs the RFC 9421 signature base from request components
4. The library hashes with EIP-191 prefix and calls `signMessage`
5. The library attaches `Signature-Input`, `Signature`, and `Content-Digest` headers

```typescript
import { createSignerClient } from '@slicekit/erc8128'

const signer = {
  chainId: 8453,
  address: tzAccountAddress,  // TZ Account — the smart account
  signMessage: async (message: Uint8Array) => {
    // Agent's EOA signs, but keyid points to the TZ Account
    return walletClient.signMessage({
      account: agentEOA,
      message: { raw: message }
    })
  }
}

const client = createSignerClient(signer, { ttlSeconds: 60 })
const response = await client.fetch('https://data-api.example.com/market-data')
```

### Server side (verification + authorization)

```
Request arrives with Signature-Input, Signature, Content-Digest headers
  │
  ├── 1. ERC-8128 Auth Middleware
  │     Parse Signature-Input → extract keyid, nonce, timestamps
  │     Parse keyid → chainId=8453, address=0xTZAccount
  │     Reconstruct signature base from request
  │     Call verifyRequest() from @slicekit/erc8128
  │       → publicClient.verifyMessage() internally calls
  │         isValidSignature(hash, sig) on TZ Account (ERC-1271)
  │       → TZ Account routes to HatValidator
  │       → HatValidator recovers signer, checks hat-wearing
  │       → Returns 0x1626ba7e if valid
  │     If invalid → 401 Unauthorized
  │     Attach { address, chainId } to request context
  │
  ├── 2. Permission Middleware
  │     Read zoneAccount from request context
  │     Look up required permission token ID for this endpoint
  │     Call registry.balanceOf(zoneAccount, permTokenId)
  │     If balance == 0 → 403 Forbidden
  │     Fetch token metadata for logging
  │     Attach { tokenId, metadata } to request context
  │
  ├── 3. Route Handler
  │     Serve the requested data
  │
  └── 4. Receipt Middleware (post-response)
        Build action receipt: { actor, endpoint, timestamp, ... }
        Log to Bonfires context graph (Tier 2)
```

## Middleware architecture for Express

### Auth middleware (ERC-8128 signature verification)

```typescript
import { verifyRequest, type NonceStore } from '@slicekit/erc8128'
import { createPublicClient, http } from 'viem'
import { base } from 'viem/chains'

// In-memory nonce store (sufficient for hackathon)
class MemoryNonceStore implements NonceStore {
  private consumed = new Map<string, number>()

  async consume(key: string, ttlSeconds: number): Promise<boolean> {
    const now = Date.now()
    // Clean expired
    for (const [k, exp] of this.consumed) {
      if (exp < now) this.consumed.delete(k)
    }
    if (this.consumed.has(key)) return false
    this.consumed.set(key, now + ttlSeconds * 1000)
    return true
  }
}

const publicClient = createPublicClient({ chain: base, transport: http(RPC_URL) })
const nonceStore = new MemoryNonceStore()

async function verifyERC8128(req, res, next) {
  // Convert Express request to Fetch API Request
  const url = `${req.protocol}://${req.get('host')}${req.originalUrl}`
  const fetchRequest = new Request(url, {
    method: req.method,
    headers: req.headers,
    body: req.method !== 'GET' ? JSON.stringify(req.body) : undefined,
  })

  const result = await verifyRequest({
    request: fetchRequest,
    verifyMessage: publicClient.verifyMessage,  // handles EOA + ERC-1271
    nonceStore,
    policy: {
      maxValiditySec: 120,
      clockSkewSec: 10,
    },
  })

  if (!result.ok) {
    return res.status(401).json({
      error: 'Unauthorized',
      reason: result.reason,  // e.g. 'expired', 'bad_signature', 'replay'
    })
  }

  // Attach verified identity to request
  req.auth = {
    address: result.address,   // TZ Account address
    chainId: result.chainId,   // 8453
  }
  next()
}
```

Key detail: `publicClient.verifyMessage` from viem already handles ERC-1271 for smart contract accounts. When the address is a contract, viem calls `isValidSignature()` on it. No custom ERC-1271 plumbing needed.

### Permission middleware (resource token balance check)

```typescript
async function checkPermission(req, res, next) {
  const { address: zoneAccount } = req.auth

  // Map endpoint to required permission token ID
  const tokenId = config.endpointTokenMap[req.path]
  if (!tokenId) {
    return res.status(403).json({ error: 'No permission token configured for this endpoint' })
  }

  const balance = await publicClient.readContract({
    address: config.resourceTokenRegistry,
    abi: ResourceTokenRegistryABI,
    functionName: 'balanceOf',
    args: [zoneAccount, tokenId],
  })

  if (balance === 0n) {
    return res.status(403).json({
      error: 'Missing permission token',
      zoneAccount,
      requiredToken: tokenId.toString(),
    })
  }

  req.permission = { tokenId }
  next()
}
```

### No directive middleware

Directives are **not deterministically enforceable** by definition — their enforcement is always post-hoc via the adjudicator. The data API does NOT check directives. If data was accessed legitimately (valid auth + permission token), it's served. Whether the agent then violates a directive (e.g., redistribution) is detected and adjudicated later.

This is a core design principle: constraints are enforced at access time, directives are enforced after the fact.

### Receipt generation middleware (post-response)

```typescript
async function logReceipt(req, res, next) {
  // Run after response is sent (use res.on('finish'))
  const originalEnd = res.end
  res.end = function(...args) {
    originalEnd.apply(res, args)

    const receipt = {
      actor: req.auth.address,
      chainId: req.auth.chainId,
      endpoint: req.path,
      method: req.method,
      permissionTokenId: req.permission?.tokenId?.toString(),
      statusCode: res.statusCode,
      timestamp: Date.now(),
    }

    // Fire-and-forget to Bonfires
    logToBonfires(receipt).catch(err =>
      console.error('[receipt] Failed to log:', err)
    )
  }
  next()
}
```

## How TZ Accounts sign requests

The signing flow bridges the agent's EOA to the TZ Account identity:

1. Agent holds an EOA private key
2. Agent wears the TZ Account's hat (via Hats Protocol)
3. Agent creates a signer where `address` = TZ Account address, but `signMessage` uses the EOA key
4. `@slicekit/erc8128` builds the signature base and calls `signMessage`
5. The resulting signature is an EOA signature, but the `keyid` points to the TZ Account
6. Server calls `isValidSignature(hash, sig)` on the TZ Account
7. TZ Account's HatValidator recovers the EOA from the signature, checks it wears the hat
8. If the EOA wears the hat, returns `0x1626ba7e` (valid)

### Integration with SDK's `signAsZone()`

The existing SDK function `signAsZone()` in `packages/sdk/src/zone.ts` provides a lower-level version of this flow. It produces an `ERC8128SignedRequest` with `keyId`, `signature`, and `message` fields but does not construct the full RFC 9421 headers.

For the real implementation, we should either:

**Option A (recommended):** Use `@slicekit/erc8128` directly in the data API client code. The SDK's `signAsZone()` becomes a helper for constructing the signer object:

```typescript
import { createSignerClient } from '@slicekit/erc8128'
import { signAsZone } from '@trustzones/sdk'

// Build a @slicekit/erc8128 signer from an SDK wallet client
function createZoneSigner(walletClient, zoneAccount, chainId) {
  return {
    chainId,
    address: zoneAccount,
    signMessage: async (message) => {
      return walletClient.signMessage({
        account: walletClient.account,
        message: { raw: message },
      })
    },
  }
}

const signer = createZoneSigner(walletClient, tzAccountAddress, 8453)
const client = createSignerClient(signer, { ttlSeconds: 60 })
```

**Option B:** Wrap `@slicekit/erc8128` inside the SDK, exposing a `createZoneClient()` that handles signer construction. Cleaner API surface but more coupling.

For hackathon: Option A. Keep the SDK thin. Use `@slicekit/erc8128` directly.

## SDK distribution for agents

Option A (recommended) has the SDK used locally by agents to sign ERC-8128 requests. This means agents need access to the SDK. Two distribution paths:

### Option 1: CLI with companion skill

Publish `@trust-zones/sdk` as an npm package with a CLI entry point:

```bash
npx @trust-zones/sdk sign --zone 0x1234 --endpoint /market-data --host api.example.com
# outputs: headers to attach to request
```

Agents (especially LLM agents) can use this via a companion skill/tool definition that wraps the CLI.

### Option 2: MCP server

Expose the SDK as an MCP server that agents can connect to:

```typescript
// MCP tools exposed:
// - signRequest({ zoneAddress, method, path, host }) → { headers }
// - verifyRequest({ headers, body }) → { address, chainId }
// - encodePropose({ schemaDoc }) → { inputId, payload }

// - compileSchemaDoc({ doc }) → { proposalData }
// - decompileProposalData({ data }) → { schemaDoc }
```

This is cleaner for AI agents — they get typed tool definitions. The MCP server wraps both `@trust-zones/sdk` and `@trust-zones/compiler`.

### Recommendation

For hackathon: **Option 2 (MCP server)** if we have time, **Option 1 (CLI)** as minimum viable. The MCP server is more impressive for demos and judge evaluation, but the CLI works today with zero infrastructure.

Both options assume agents install/connect locally. The signing happens on the agent's machine using their private key — the SDK never sees remote keys.

## Security considerations

1. **Replay attacks**: Non-replayable mode (default) with nonce tracking prevents replays. The `MemoryNonceStore` is sufficient for a single-server hackathon demo but would need Redis/distributed storage in production.

2. **Clock skew**: Set `clockSkewSec: 10` to tolerate minor time differences. Keep `maxValiditySec` short (60-120s) to limit the window for intercepted signatures.

3. **Request binding**: Request-bound signatures prevent an intercepted signature from being used against a different endpoint or with different parameters. Always use request-bound mode for data APIs.

4. **Chain verification**: The server must verify against the correct chain. Parse `chainId` from the keyid and use the corresponding RPC endpoint. For our hackathon: hardcode Base (8453).

5. **RPC trust**: The server trusts its RPC provider for `isValidSignature()` and `balanceOf()` calls. A compromised RPC could return false positives. In production, use multiple RPC endpoints or a local node.

6. **Permission token freshness**: Token balances are read at request time, so revoked permissions take effect immediately. However, onchain reads add latency (~100-500ms per call). Consider caching with short TTLs (30-60s) for non-critical reads.

7. **Rate limit state**: In-memory rate limit counters reset on server restart. For the hackathon this is acceptable. The authoritative record for post-hoc adjudication is the receipt log in Bonfires, not the server's internal counters.

8. **Content-Digest**: For requests with bodies (POST), the `Content-Digest` header prevents body tampering. The `@slicekit/erc8128` library handles this automatically.

## Open questions

1. **Express request conversion**: `@slicekit/erc8128` expects a Fetch API `Request` object. Express provides its own request format. The conversion (constructing a `new Request()` from Express's `req`) must faithfully reproduce all headers. Need to verify this works correctly with the library, especially for `Content-Digest` on POST bodies.

2. **viem `verifyMessage` and ERC-1271**: viem's `publicClient.verifyMessage()` should handle ERC-1271 automatically when the address is a contract. Need to verify this is the case and that it works with the `@slicekit/erc8128` library's expected callback signature.

3. **Hono vs Express**: `@slicekit/erc8128` provides a Hono middleware example natively. Hono uses standard Fetch API `Request` objects, avoiding the Express conversion issue. Consider switching the data API server from Express to Hono.

4. **ERC-6492 support**: ERC-6492 enables signature validation for pre-deploy contracts (counterfactual accounts). Not needed for hackathon since TZ Accounts are deployed before agents use them, but worth noting for production.
